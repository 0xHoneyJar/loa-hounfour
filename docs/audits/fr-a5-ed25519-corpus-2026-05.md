# FR-A5 Consumer-Corpus Audit — Ed25519 Pattern Narrowing

> **Load-bearing evidence (TL;DR):** **Ed25519 signatures are exactly
> 64 bytes, encoding to exactly 86 unpadded base64url characters (RFC
> 4648 §5: `ceil(64 * 8 / 6) = 86`, no remainder, no `=` padding). The
> `{87,88}` forms the schemas previously accepted are mathematically
> unreachable for any spec-conformant signer.** This cryptographic-
> impossibility argument is the headline; the consumer-corpus survey
> below is corroborative. Even an incomplete survey (one repo not
> present at audit time, see Limitations §1) does not weaken the
> classification because the surveyed-out range cannot exist in the
> wild for any spec-conformant producer.

## Why this audit gates the bump

A regex narrowing is, in the abstract, a strict-non-additive change. Under
strict semver it would require a major bump. The constitutional rule the
project follows is: **"narrowing is breaking unless the narrowed-out range
is empty across surveyed consumers."** This audit produces the surveyed-
empty evidence — it queries every consumer repo on file at audit time
for any payload whose signature would no longer validate under the
narrower pattern.

The classification language is deliberately "surveyed-empty" rather than
"provably empty" — the audit covers the consumers it can reach (see
**Surveyed consumer repos** below for the N-of-known-N count and the
limitations section for the residual). A *cryptographic* completeness
argument is also available and reinforces the surveyed result:
**Ed25519 signatures are exactly 64 bytes**, which encodes to **exactly
86 unpadded base64url characters** (RFC 4648 §5; `ceil(64 * 8 / 6) = 86`,
no remainder so no `=` padding). The `{87,88}` forms the schemas
previously accepted are **mathematically unreachable** for any
spec-conformant signer. The surveyed-empty result and the
cryptographic-impossibility result together gate the change.

If zero hits + cryptographic-unreachability: the narrowing reduces the
legal-input surface area without invalidating any extant payload (and
without invalidating any spec-conformant future payload). The change
ships under a minor bump with this audit committed as the auditable
record. If any hits: the work is deferred to v9.0.0 and the audit
document records the deferral cause.

## Method

**Two-pass grep** per consumer repo:

1. **Extension-restricted pass** (sanity check): `*.ts`, `*.tsx`, `*.js`, `*.json`, `*.md`, `*.yaml`, `*.yml`.
2. **Broad sweep** (the binding form): every text file regardless of extension.

Both passes exclude vendor / build / VCS / cache directories and `*.lock` /
`*.log` files so generated artifacts don't pollute the result.

### Binding-form command

The broad-sweep command is the audit of record. The extension-restricted
form is run first as a fast pre-check; if the broad sweep returns more
hits than the extension form, that itself is a flag (file-extension drift
in the consumer corpus).

```bash
grep -rEn --binary-files=without-match 'ed25519:[A-Za-z0-9_-]{87,88}' "$repo" 2>/dev/null \
  | grep -vE '/(node_modules|dist|build|\.git|\.next|coverage|\.turbo|\.cache|target|out)/' \
  | grep -vE '\.(lock|log)$'
```

The pattern `[A-Za-z0-9_-]{87,88}` is the regex quantifier sense — it
matches strings of 87 OR 88 base64url characters (i.e. the range that
the schema currently permits but the FR-A5 narrowing removes).

## Reproducing the audit

To re-run on demand (staleness check, new consumer onboarded, post-incident
verification). The reproducer parameterizes the consumer-repo root via
`THJ_REPOS_ROOT` (defaulting to the path used at original audit time) so
the same script can run on CI or another contributor's machine without
edits — the historical absolute paths in the table above remain the
audit-of-record.

```bash
# THJ_REPOS_ROOT defaults to the path used at original audit time; on a
# different machine, override (e.g. THJ_REPOS_ROOT=$(dirname "$PWD") if
# this repo and its sibling consumer repos share a parent directory).
THJ_REPOS_ROOT="${THJ_REPOS_ROOT:-$HOME/Documents/thj/code}"
for name in loa-finn loa-dixie loa-freeside; do
  repo="$THJ_REPOS_ROOT/$name"
  [ -d "$repo" ] || { printf '  %-50s → SKIPPED (not present)\n' "$name"; continue; }
  hits=$(grep -rEn --binary-files=without-match 'ed25519:[A-Za-z0-9_-]{87,88}' "$repo" 2>/dev/null \
           | grep -vE '/(node_modules|dist|build|\.git|\.next|coverage|\.turbo|\.cache|target|out)/' \
           | grep -vE '\.(lock|log)$' \
           | wc -l)
  printf '  %-50s → %d hits\n' "$name" "$hits"
done
```

Expected output (matching the audit table; absent repos report SKIPPED):

```
  loa-finn                                           → 0 hits
  loa-dixie                                          → 0 hits
  loa-freeside                                       → 0 hits
```

## Limitations

1. **`loa-oracle` not surveyed** — the repo was not present locally at
   audit time. Per sprint plan §1 PR-A3.1 Step 1's "if present; check
   first — may not exist locally" clause, the repo is skipped. If
   `loa-oracle` lands during the v8.6.x line and surfaces a real `{87,88}`
   payload, the rollback path applies.
2. **Local-clone snapshot only** — the audit greps the local working
   trees of consumer repos, not their full git history. A deleted-but-
   historic `{87,88}` payload would not surface. The constitutional
   question is what consumers ARE running, not what they once ran;
   working-tree state is the right surface.
3. **Same-day timestamp** — this audit was run on the same day as the
   PR-A3.1 implementation. Time-since-audit staleness is not a concern
   for this PR. If meaningful time elapses before any future use of
   this audit (e.g. re-evaluation in a v9.0.0 RFC), re-run.
