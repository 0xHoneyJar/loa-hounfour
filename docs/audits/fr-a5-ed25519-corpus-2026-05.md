## Why this audit gates the bump

A regex narrowing is, in the abstract, a strict-non-additive change. Under
strict semver it would require a major bump. The constitutional rule the
project follows is: **"narrowing is breaking unless the narrowed-out range
is provably empty in the wild."** This audit is the proof of empty — it
queries every consumer repo on file at audit time for any payload whose
signature would no longer validate under the narrower pattern.

If zero hits: the narrowing reduces the legal-input surface area without
invalidating any extant payload. The change ships under a minor bump with
this audit committed as the auditable record. If any hits: the work is
deferred to v9.0.0 and the audit document records the deferral cause.

## Method

**Two-pass grep** per consumer repo:

1. **Extension-restricted pass** (sanity check): `*.ts`, `*.tsx`, `*.js`, `*.json`, `*.md`, `*.yaml`, `*.yml`.
2. **Broad sweep** (the binding form): every text file regardless of extension.

Both passes exclude vendor / build / VCS / cache directories and `*.lock` /
`*.log` files so generated artifacts don't pollute the result.

### Binding-form command

The broad-sweep command is the audit of record. The extension-restricted
form is run first as a fast pre-check; if the broad sweep returns more
hits than the extension form, that itself is a flag (file-extension drift
in the consumer corpus).

```bash
grep -rEn --binary-files=without-match 'ed25519:[A-Za-z0-9_-]{87,88}' "$repo" 2>/dev/null \
  | grep -vE '/(node_modules|dist|build|\.git|\.next|coverage|\.turbo|\.cache|target|out)/' \
  | grep -vE '\.(lock|log)$'
```

The pattern `[A-Za-z0-9_-]{87,88}` is the regex quantifier sense — it
matches strings of 87 OR 88 base64url characters (i.e. the range that
the schema currently permits but the FR-A5 narrowing removes).

## Limitations

1. **`loa-oracle` not surveyed** — the repo was not present locally at
   audit time. Per sprint plan §1 PR-A3.1 Step 1's "if present; check
   first — may not exist locally" clause, the repo is skipped. If
   `loa-oracle` lands during the v8.6.x line and surfaces a real `{87,88}`
   payload, the rollback path applies.
2. **Local-clone snapshot only** — the audit greps the local working
   trees of consumer repos, not their full git history. A deleted-but-
   historic `{87,88}` payload would not surface. The constitutional
   question is what consumers ARE running, not what they once ran;
   working-tree state is the right surface.
3. **Same-day timestamp** — this audit was run on the same day as the
   PR-A3.1 implementation. Time-since-audit staleness is not a concern
   for this PR. If meaningful time elapses before any future use of
   this audit (e.g. re-evaluation in a v9.0.0 RFC), re-run.
