# v8.5.0 Migration — Class-Validation Schema Intake

> **Status:** v8.5.0 (in flight; PR-A2.1 lands this doc; PR-A2.2 / PR-A2.3 ship the schemas)
> **See also:** [ADR-010](../adr/ADR-010-class-vs-policy-boundary.md), `docs/architecture/hashing-spec-freeze-v8.5.md`

A consumer-facing class-validation surface arrived alongside cycle-004
with ~50 candidate types. v8.5.0 reuses the substrate-agnostic core,
extends a small set of existing `$id`s additively, adds the
authority-cascade + recall + forget + commit + assertion primitives,
defers a Challenge layer and an EstateTransition family to cycle-005,
and routes one closed enum to a vocabulary file rather than a `$id`.

This document mirrors PRD Appendix A row by row and explains the
*why* behind each decision: the boundary, the substrate-agnosticism
filter, the FOLD rationale, and the cycle-005 deferral logic.

| # | Intake name | Decision | Target | Rationale |
|---|---|---|---|---|
| 1 | `Actor` | REUSE | `AgentIdentity` | Concept maps; `actor_id` ↔ `agent_id`, `actor_type` ↔ `agent_type`. |

**Why REUSE rather than coin new names**: each of the three identity-
layer `$id`s already exists in the v8.4.0 surface and carries the same
semantics under a different name. Adding parallel `$id`s would either
duplicate the surface (consumer ergonomics fall) or split it (downstream
schemas fork between `Actor` and `AgentIdentity`). The intake renames
at its own boundary; hounfour stays single-source.

### CandidateAssertion fold (F3) — rationale

The intake ships `CandidateAssertion` as a parallel type: same body
fields, no signatures, distinct lifecycle. The fold makes it a status
on `Assertion` (`status: 'candidate'`) rather than a separate `$id`.
Two reasons:

1. **Single object as it transitions.** A candidate becoming admitted
   is a status change, not a type change. With a fold, the same
   object instance walks `candidate → admitted → superseded`; without,
   the consumer hand-translates between two parallel shapes.
2. **Discriminator-aware crypto-bearing flag (J3).** The `Assertion`
   schema is crypto-bearing only when `status` is in the
   non-candidate set. Encoding that as a per-variant TypeBox metadata
   flag (rather than two parallel `$id`s) keeps the variant analysis
   uniform with `validate()` runtime + lint detection.

## Substrate-agnosticism filter

When an intake candidate uses a wedge-fitted name, hounfour adopts a
substrate-agnostic equivalent. The filter is mechanical:

1. **The protocol does not name its consumers in primitives.** Names
   like `EnvironmentFrame` (carrying intake-specific environment
   semantics) become `SurfaceContext` (generic 5-member core).
2. **Closed enums with consumer-specific members are split.** A
   16-member `AssertionClass` becomes a 6-8-member substrate-agnostic
   core; the residual members live in consumer-side namespace
   extensions. Same recipe for `AssertionStatus`, `ProvenanceSourceType`.
3. **Open dotted-string vocabularies stay open.** `AuditEvent.event_type`
   and the new audit-event-types vocabulary file ride the
   `<github-org>:<consumer>:<event_type>` 3-segment namespace pattern;
   hounfour ships a soft prefix registry (`AUDIT_EVENT_TYPES_KNOWN_PREFIXES`)
   without arbitrating uniqueness.

Two CapabilityScopedTrust EXTEND fields (OQ-S6) and the AssertionClass
core member set (OQ-S3) are the two design-freeze decisions resolved
in PR-A2.1's NOTES.md Decision Log.

## Vocabulary harmonization (Phase 4 follow-up)

`OrgRepresentativeDelegation.capability_scope` already exists; v8.5.0
adds an ORD-5 sub-rule tying that field to the canonical
`CapabilityScope` vocabulary that Layer 2 (`SignerEntry.scoped_trust`)
and Layer 3 (`SignerCompetenceRule.required_capability_scopes`) both
consume. The sub-rule lands in warning-mode initially per R3 (vocabulary
harmonization risk); cycle-005 escalates to error-mode after a soak
window.

The single shared `CapabilityScope` vocabulary is the user-driven
plug-point identified in the discovery Phase 4 — without it, the
authority cascade fragments into three parallel scope languages.

## Successor mapping (cycle-005)

Six candidates DEFER explicitly; the order they re-enter the queue:

1. **Challenge layer** (`Challenge`, `ChallengeType`, `ChallengeRequestedEffect`)
   — paired with the cross-runner extension that's also a cycle-005
   deliverable.
2. **EstateTransition family** (`EstateTransition`, `TransitionReceipt`,
   `TransitionReceiptKind`) — designed alongside the Challenge layer
   so transitions can carry challenge outcomes.
3. **`AssertionClass` member-set escalation** — if the v8.5.0 6-8-member
   core proves too narrow during the rc.1 shadow-integration window,
   cycle-005 expands it (additively).
4. **Recall sub-component `$id` promotion** — if W3 cut-line trips and
   `RecallItem` / `RecallUseInstruction` / `RedactionSummary` /
   `ExclusionSummary` ship inlined into `RecallPack`, cycle-005 promotes
   them to top-level `$id`s if any consumer surfaces a need.
5. **Vocabulary-harmonization escalation** — ORD-5 warn-mode → error-mode.
